Audit and trust boundary

The audit path is not a side note. It records stage-level counts, token footprints, and keep/drop/compress decisions so a developer can debug why the final prompt looks the way it does.

Verification surface

The tests mirror the architecture: optimizer, ranking, deduplication, MMR, budgeter, config, dashboard, models, recipes, and prompt builder each have targeted coverage.

Review targets for production use
  • Model dependency weight: embedding and reranking quality come with model download size, runtime cost, and CPU/GPU variability.
  • Compression trust: README warns that compression can preserve facts while breaking answerability, so default strategies rely on selection/dedup first.
  • Benchmark scope: treat reported savings as representative until validated on the target app's retrieval corpus.
  • Audit persistence: final prompt and audit report have different privacy needs.
  • Config drift: thresholds and strategies should be tracked with evaluation runs.
