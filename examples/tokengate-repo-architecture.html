tokengate / repository overview

Context optimization library for LLM applications

TokenGate replaces top-k prompt stuffing with a staged context pipeline. It ranks, reranks, deduplicates, selects, budgets, optionally compresses, builds the final prompt, and records every decision in an audit trail.

Python 3.12

Package metadata pins the runtime for ML-wheel reliability.

No LLM in pipeline

README states optimization runs offline and is app-agnostic.

Typed package

`py.typed`, strict mypy config, and public exports indicate typed API intent.

MIT

The source repository is public and open source.

Evidence source: public README, pyproject metadata, module tree, and architecture notes. This visual avoids local secrets, env values, and raw private source dumps.
src / tokengate

Package boundary

The package exposes a compact public API while keeping ranking, selection, deduplication, budgeting, audit, dashboard, and model integrations in separate modules.

Public entrypoint

`__init__.py` exports TokenGate, TokenBlock, OptimizerConfig, result models, token counters, and error types.

Stage separation

Each pipeline concern has its own folder, making the architecture readable from the tree.

Secret-safe read

No credential values are needed to explain the package architecture.

Core contract layer

Core owns the main optimizer object, configuration, token block model, and result/audit data structures. Other folders plug into this contract.

Coordinates the stages and returns an OptimizationResult.

config.py

Represents candidate chunks with scores, metadata, and token counts.

result.py

Captures included, compressed, dropped blocks and audit records.

src/tokengate/core/optimizer.py

Pipeline coordinator

This is the file a reader should inspect first when they want the end-to-end flow. The visual summary treats it as the orchestration layer that calls ranking, deduplication, selection, budgeting, prompt building, and audit generation.

Input

Query plus candidate TokenBlock objects.

Process

Run staged context optimization under a selected strategy.

Output

Final prompt, kept blocks, dropped blocks, compressed blocks, audit trail.

src/tokengate/core/config.py

Configuration and strategy surface

Config makes TokenGate tunable without changing application code. README examples show a `tokengate.toml` path and environment-based config loading.

Strategies

speed, balanced, quality, and max_compression represent different tradeoffs.

Budgets

Prompt budget and headroom protect downstream context limits.

Thresholds

Relevance floor, dedup thresholds, MMR lambda, and source priorities control selection.

Validation

README says unknown config keys raise, so typos should not silently no-op.

src/tokengate/core/result.py

Result and audit data models

The result model is the product promise: not just a shorter prompt, but a trace of what changed and why.

Included

Blocks that survived to the final prompt.

Compressed

Blocks shortened by an opt-in compression path.

Dropped

Blocks rejected by duplicate, score, diversity, or budget logic.

src/tokengate/ranking

Ranking stage

Ranking combines multiple signals before final selection. The module names show semantic scoring, keyword ranking, hybrid ranking, normalization, reranking, and auxiliary signals.

semantic

Embedding similarity from model-backed vectors.

keyword

Lexical matching for direct signal recovery.

reranker

Cross-encoder scoring for query/chunk pairs.

signals

Recency, token efficiency, and source priority.

Hybrid ranker

Combines ranking signals into a unified score before later stages apply adaptive cutoff, deduplication, diversity, and budget constraints.

src/tokengate/ranking/reranker_stage.py

Cross-encoder reranking stage

The README describes BGE reranking after initial hybrid ranking. This file is the likely stage boundary for applying reranker relevance scores.

Purpose

Improve ranking quality using query/chunk pair scoring.

Tradeoff

Better relevance costs more runtime than pure keyword or embedding scoring.

src/tokengate/selection

Selection stage

Selection narrows high-ranked candidates using per-query adaptive cutoff and MMR diversity. This is where the system stops treating all top-N chunks as equally useful.

adaptive_cutoff.py

Finds the relevance cliff for the current query.

mmr.py

Balances relevance and diversity in the final candidate set.

src/tokengate/deduplication

Deduplication stage

TokenGate's main savings story comes from selecting and deduplicating context, not blindly compressing everything.

exact.py

Removes byte-identical content.

near_duplicate.py

Handles close textual overlap.

semantic.py

Collapses near-paraphrases with vector similarity.

src/tokengate/budgeting

Budgeting stage

The budgeter decides which selected blocks fit the prompt budget. The README frames this as value-per-token fitting with token counters that can be heuristic or model-aware.

budgeter.py

Fits required and optional blocks under the context budget.

token_counter.py

Supports dependency-free estimation and optional exact counting.

src/tokengate/compression

Compression is present, but intentionally off by default

The README makes a strong product decision: aggressive compression can preserve fact strings while breaking answerability, so default strategies rely on selection and deduplication instead.

extractive.py

Conservative compression path.

llmlingua_compressor.py

Optional learned compression backend.

structured_schema.py

Schema support for structured compression outputs.

src/tokengate/audit

Audit trail

Audit is the trust layer. It explains each stage's block and token counts, and every per-block keep/drop/compress decision.

dashboard sync

Review targets

These are the places a reviewer should inspect first before depending on TokenGate in a production RAG app.

Model dependency weight

BGE embedding/reranking quality comes with model download size, runtime cost, and CPU/GPU variability.

Compression trust

The README correctly warns that compression can preserve facts while breaking answerability.

Benchmark scope

Benchmark claims are useful but source-labeled as representative, not universal SOTA.

Audit persistence

Audit data should avoid storing sensitive retrieved content when used in private applications.

Configuration drift

Strategies and thresholds should be versioned for reproducible evaluations.

Integration contract

Downstream apps should treat final_prompt and audit as separate outputs with different privacy needs.
